# Bastion Conversion Design

## Goal

Turn `gosshd` from a UUID-based SSH relay into a small bastion system with SQLite-backed users, organizations, SSH public-key authentication, target aliases, command auditing, command policy groups, optional LLM command review, and an engineered management UI. The delivered branch must include complete end-to-end coverage for the core login, configuration, SSH routing, audit, and policy flows.

## Existing System

The current server is a compact Go application. `cmd/gosshd-server` starts one HTTP listener and one SSH listener. HTTP serves health checks, agent run scripts, agent binary downloads, and the agent websocket. The SSH server currently has `NoClientAuth` enabled and treats `conn.User()` as a UUID. If that UUID is online in the in-memory `AgentRegistry`, SSH session, SFTP, and TCP streams are bridged to the matching agent over yamux.

The current agent already supports three useful stream types:

- `exec` and `shell`, which run commands on the agent host.
- `sftp`, which starts an SFTP server rooted at the agent root.
- `tcp`, which dials a host and port reachable from the agent host.

The bastion design keeps the single-binary deployment model and reuses the agent TCP stream for private-network SSH targets.

## Approved Approach

Use a single Go backend with SQLite and an embedded frontend. `gosshd-server` remains the deployable server process. It will serve:

- The management web UI and JSON API.
- Existing agent download assets.
- Owner-scoped agent enrollment scripts.
- The agent websocket.
- The public SSH bastion endpoint.

The management UI uses local email/password accounts and session cookies. OAuth and SSO are out of scope for this first version.

## Core Concepts

Users represent human operators. A user can own personal SSH targets and can belong to organizations.

Organizations group users and SSH targets. A user can create an organization and can join one through an invitation code generated by an organization member with admin permissions.

Each organization has user groups. Creating an organization automatically creates a default all-members user group. Joining an organization automatically adds the user to that default group. Organization admins can create more user groups and add or remove members. Command policies can bind to one or more user groups so a policy applies only when the SSH target and authenticated user both match the policy scope.

Public keys authenticate SSH users. During SSH login, the server maps the presented public key fingerprint to a platform user. The SSH username is not the platform user name; it is the target alias. For example, in `ssh test2@public-ip`, `test2` is resolved as an SSH target alias visible to the authenticated user.

SSH targets represent remote SSH services. A target belongs either to a user or to an organization and has a unique alias within that owner scope. Targets support two connection modes:

- `direct`: the bastion server opens an SSH client connection to `host:port`.
- `agent`: the bastion server opens a TCP stream through an enrolled agent, then starts an SSH client connection to the private `host:port` reachable from that agent.

Command policies attach to one or more targets and may attach to one or more organization user groups. A policy contains blacklist and whitelist rules and may optionally delegate unmatched commands to an LLM provider configuration. A policy without user-group bindings applies to all users who can access the target. A policy with user-group bindings applies only when the authenticated user belongs to at least one bound group.

Audit logs record SSH attempts, executed commands, policy decisions, and session outcomes.

## SQLite Storage

Use `database/sql` with a Go SQLite driver. The implementation should prefer a pure-Go or low-friction driver for reliable local e2e execution on Windows and Linux. The schema is applied on server startup.

Required tables:

- `users`: id, email, display name, password hash, created time.
- `sessions`: id, user id, token hash, expiry, created time.
- `organizations`: id, name, slug, owner user id, created time.
- `organization_members`: organization id, user id, role, created time.
- `organization_user_groups`: id, organization id, name, slug, is default, created time.
- `organization_user_group_members`: group id, user id, created time.
- `organization_invites`: id, organization id, code hash, role, expiry, created by, created time, consumed time.
- `user_public_keys`: id, user id, name, authorized key text, fingerprint, created time.
- `ssh_targets`: id, owner type (`user` or `organization`), owner id, alias, target type (`direct` or `agent`), host, port, remote username, auth type (`password` or `private_key`), encrypted secret material, agent id, created by, created time, updated time.
- `agent_enrollments`: id, owner type, owner id, token hash, label, default host, default port, created by, created time, expiry, consumed agent id.
- `agents`: id, owner type, owner id, enrollment id, label, current runtime id, last seen, created time.
- `command_policies`: id, owner type, owner id, name, default action (`allow` or `deny`), llm config id, created time.
- `policy_rules`: id, policy id, rule type (`blacklist` or `whitelist`), pattern type (`exact`, `prefix`, or `contains`), pattern, created time.
- `policy_targets`: policy id, target id.
- `policy_user_groups`: policy id, group id.
- `llm_policy_configs`: id, owner type, owner id, name, base url, api key encrypted, model, prompt, timeout seconds, created time.
- `command_audit_logs`: id, user id, target id, organization id nullable, session id, command, request type, policy decision, policy reason, exit code nullable, started time, ended time, remote address.

Secret material is stored encrypted with a server key configured by flag or environment variable. If no key is configured in development, the server may generate a local key file next to the SQLite database and log that it is local-only.

## HTTP Backend

Add JSON API routes under `/api`. The API returns typed JSON errors and requires a session cookie for all routes except registration, login, logout, health, static assets, downloads, and agent enrollment script retrieval.

Authentication routes:

- `POST /api/auth/register`: create a user and session.
- `POST /api/auth/login`: create a session.
- `POST /api/auth/logout`: delete the current session.
- `GET /api/me`: return current user, organizations, and active organization context.

Organization routes:

- `POST /api/orgs`: create an organization and make the creator owner.
- `GET /api/orgs`: list organizations visible to the user.
- `POST /api/orgs/{id}/invites`: create an invite code.
- `POST /api/orgs/join`: join with an invite code.
- `GET /api/orgs/{id}/groups`: list organization user groups.
- `POST /api/orgs/{id}/groups`: create an organization user group.
- `POST /api/orgs/{id}/groups/{group_id}/members`: add a user to a group.
- `DELETE /api/orgs/{id}/groups/{group_id}/members/{user_id}`: remove a user from a group.

Key routes:

- `GET /api/keys`: list current user's public keys.
- `POST /api/keys`: add an OpenSSH authorized key. Store its normalized text and fingerprint.
- `DELETE /api/keys/{id}`: remove a key owned by the current user.

Target routes:

- `GET /api/targets?owner_type=&owner_id=`: list visible targets.
- `POST /api/targets`: create a direct or agent target in a personal or organization scope.
- `PATCH /api/targets/{id}`: update alias, host, port, remote username, auth method, and secret material.
- `DELETE /api/targets/{id}`: remove a target.

Agent routes:

- `POST /api/agent-enrollments`: create an owner-scoped enrollment token and return install commands.
- `GET /install/{token}.sh` and `GET /install/{token}.ps1`: return scripts that launch the agent with the enrollment token.
- `GET /api/agents`: list agents visible to the user.

Policy routes:

- `GET /api/policies`: list policies.
- `POST /api/policies`: create a policy with default action and optional LLM config.
- `POST /api/policies/{id}/rules`: add whitelist or blacklist rules.
- `POST /api/policies/{id}/targets`: attach targets.
- `DELETE /api/policies/{id}/targets/{target_id}`: detach targets.
- `POST /api/policies/{id}/user-groups`: attach organization user groups.
- `DELETE /api/policies/{id}/user-groups/{group_id}`: detach organization user groups.

Audit routes:

- `GET /api/audit`: list command audit logs visible to the user, filterable by owner, target, user, and time range.

## Frontend

Create an engineered frontend under `web/` and embed its production build into the Go server. The UI should be operational rather than marketing-oriented.

Primary screens:

- Login and registration.
- Organization switcher with create organization and join organization flows.
- Organization user group management, including the default all-members group.
- Public key management.
- SSH target list with personal and organization scopes.
- Target create/edit form supporting direct and agent-backed targets.
- Agent enrollment view showing copyable Linux/macOS and Windows install commands.
- Command policy editor with whitelist, blacklist, default action, and LLM settings.
- Audit log table with target, user, command, decision, exit code, and timestamps.

The UI should be built with predictable dense layouts: top navigation, left sidebar or tabbed management sections, tables for targets and audit logs, compact forms, and clear status badges.

## SSH Routing

Replace `NoClientAuth` with public-key authentication. The SSH server should use `PublicKeyCallback` to compute the key fingerprint and look up a user from SQLite.

On successful authentication, store the authenticated platform user id in SSH connection permissions. Continue using `conn.User()` as the alias. For `ssh test2@public-ip`, `test2` is the target alias.

Target resolution order:

1. Personal target for the authenticated user with the requested alias.
2. Organization targets with the requested alias in organizations where the user is a member.

If multiple organization targets match the same alias, reject the connection with a clear ambiguity error. The user can disambiguate by using unique aliases. The first version does not overload SSH usernames with organization prefixes.

For `exec`, `shell`, and `sftp`, the bastion opens an SSH client connection to the target and bridges channels between the user's SSH client and the target SSH server.

For `direct-tcpip`, the bastion resolves the alias once and opens matching TCP forwarding behavior through the target connection when supported. If direct TCP forwarding proves too large for the first implementation, the server must reject it explicitly and keep session, exec, and sftp fully working.

## Target Connections

Direct targets:

- Connect from the bastion server to the configured `host:port`.
- Authenticate as `remote_username` using stored password or stored private key.
- Bridge session, exec, shell, and sftp channels to the remote SSH server.

Agent targets:

- Resolve an enrolled online agent from the target.
- Open a `StreamTCP` request to the target host and port through the existing agent protocol.
- Run an SSH client handshake over that stream using the target's remote username and secret material.
- Bridge SSH channels the same way as direct targets.

This makes the previous curl-installed agent part of the bastion resource model instead of a standalone UUID login secret.

## Agent Enrollment

The existing `/run.sh` and `/run.ps1` routes remain for backward compatibility while the new owner-scoped install scripts live under `/install/{token}.sh` and `/install/{token}.ps1`.

The agent protocol adds an enrollment token field. When an agent connects with an enrollment token, the server validates it against `agent_enrollments`, creates or updates an `agents` row, and registers the runtime session under the persisted agent id. The in-memory registry should move from raw agent UUIDs to persisted agent ids while still tracking the current yamux session.

Install scripts set the server URL, public SSH hint, and enrollment token. The agent still prints a useful status message, but access instructions should point users to the management UI target alias flow rather than `ssh UUID@host`.

## Command Auditing

For `exec`, record:

- User id.
- Target id.
- Organization id if the target is organization-owned.
- SSH alias.
- Command text.
- Policy decision and reason.
- Start time.
- End time.
- Exit code.
- Remote address.

For `shell`, record session start and session end. Also record best-effort line-based input commands when input contains a newline and can be decoded as text. These line records are audit-oriented and not relied on for strict policy enforcement in the first version.

For `sftp`, record session start and end as an SFTP audit event. Per-file operation auditing is out of scope for the first version.

## Command Policy

Before executing an `exec` command, evaluate policies attached to the target and applicable to the authenticated user. A policy is applicable when it has no user-group bindings or when the user belongs to at least one bound group.

Evaluation order:

1. If a whitelist rule matches, allow.
2. If a blacklist rule matches, deny.
3. If no local rule matches and the policy has an LLM config, call the configured LLM and allow only if the response says to allow.
4. If no LLM config applies, use the policy default action.

Rule matching:

- `exact`: command equals pattern after trimming spaces.
- `prefix`: command starts with pattern after trimming spaces.
- `contains`: command contains pattern.

If multiple policies are attached, the most restrictive decision wins. Any deny rejects the command. LLM errors are treated as deny when the policy default action is deny, and as deny when the policy explicitly requires LLM review for unmatched commands.

Denying a command should write an audit log with decision `deny`, send a clear message to stderr, and return exit code 126.

## LLM Review

The first version uses an OpenAI-compatible chat completions HTTP contract configured by base URL, API key, model, and prompt. The prompt receives structured JSON with user, target, command, and matching context. The expected model output is JSON with:

- `allow`: boolean.
- `reason`: string.

The implementation must validate the response and fail closed when response parsing fails.

## Error Handling

HTTP APIs return consistent JSON errors:

```json
{"error":"message"}
```

SSH authentication failures should not reveal whether a key belongs to a known user. Target resolution errors may reveal alias availability only after authentication.

Agent offline errors should be clear in the UI and in SSH stderr.

## E2E Requirements

The delivered branch must include an automated e2e test that proves the core product flow:

1. Start the server with a temporary SQLite database and temporary host key.
2. Register a user through the HTTP API.
3. Add the user's SSH public key through the HTTP API.
4. Create an organization through the HTTP API.
5. Assert the organization has a default all-members user group and the creator belongs to it.
6. Add a direct SSH target alias `test2` using a local test SSH server.
7. Connect with an SSH client using the private key and `User: test2`.
8. Execute a command and assert the remote command output is returned.
9. Query audit logs and assert the command, user, target, decision, and exit code were recorded.
10. Attach a blacklist policy to `test2` and the default all-members user group, execute a denied command, assert exit code 126 and audit decision `deny`.
11. Attach a whitelist rule for an allowed command, execute it, and assert success.
12. Create an agent enrollment, start an agent using that token, add an agent-backed target alias, execute a command through it, and assert output plus audit log.
13. Exercise UI build artifacts by building the frontend and verifying the server serves the built app entrypoint.

The test command should be documented and runnable on the development machine. If the frontend uses Node tooling, the e2e command must install dependencies through the lockfile or use the checked-in dependency metadata.

## Testing Strategy

Unit tests:

- SQLite migration and repository behavior.
- Session creation and authentication.
- Public key normalization and fingerprint lookup.
- Target alias resolution across personal and organization scopes.
- Policy rule matching and decision precedence.
- LLM response parsing.

Integration tests:

- HTTP API auth, organization, key, target, policy, audit flows.
- Organization user group creation, default group membership, and policy user-group binding flows.
- Agent websocket enrollment and registry behavior.
- SSH public-key auth and target resolution.

E2E tests:

- The full user-visible flow described above.

Baseline `go test ./...` must pass using a reachable Go module proxy. In the current development environment, `GOPROXY=https://goproxy.cn,direct` is known to work.

## Non-Goals For First Version

- OAuth, SSO, SAML, or external identity provider integration.
- Per-file SFTP audit.
- Full terminal command parsing with shell-aware semantics.
- Multi-tenant hostname-based SSH routing.
- High availability or multi-node agent registry.
- Fine-grained RBAC beyond owner/admin/member roles.

## Implementation Boundaries

Keep code split by responsibility:

- Storage and migrations in a dedicated internal package.
- Auth/session services separate from HTTP handlers.
- Organization, key, target, policy, and audit services with repository-backed methods.
- SSH bastion routing separate from low-level SSH channel bridging.
- Agent enrollment separate from existing download script generation.
- Frontend source under `web/`, built output embedded by the Go server.

The existing relay behavior should be migrated rather than deleted abruptly. Backward-compatible agent download URLs remain available while new owner-scoped enrollment scripts become the preferred path.
